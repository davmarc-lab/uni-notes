\section{The CAP Theorem}\label{sec:the_cap_theorem} % (fold)

\subsection{Availability, Consistency, Failure}\label{sec:availability_consistency_failure} % (fold)
Being able to \textbf{keep on providing services in spite of failures} is supposed to be one of the main benefit of distributed systems over centralised ones.
\begin{note}[Intuitive Assumption]
    Distributed systems \textbf{can be designed} so that if one component of the system fails -- or, it becomes disconnected/partioned -- other components can replace it \textbf{to hide failures} from the outside world.
\end{note}
When a system can hide (most of) it failures, it is basically (almost) \textit{always working} (highly available). Basic questions in the design of distributed systems: \textit{how much failure} can a given system sustain \textit{before failure is noticed}?
\\
\begin{note}[What do we expect from a distributed system?]
    \begin{itemize}
        \item \textbf{consistency} -- we would like them to behave \textit{correctly}, to give correct replies when queried;
        \item \textbf{availability} -- we would like them to work, to make good things happen, to be \textit{live};
        \item \textbf{unreliability} -- however, we know systems \textit{can} actually experience power losses, crashes, network failures, message loss, malicious attacks, Byzantine failure, and so on;
    \end{itemize}    
\end{note}
The CAP theorem is an example of a more general tradeoff between \textit{safety} and \textit{liveness} in \textit{unreliable} systems.
% subsubsection Availability, Consistency, Failure (end)
%
\subsection{CAP at a Glance}\label{sub:cap_at_a_glance} % (fold)
\begin{theorem}{CAP Theorem}{cap_thr}
    The \textbf{CAP} theorem states that is impossible, to satisfy all three of the following guarantees, in distributed applications:
    \begin{itemize}
        \item \textbf{Consistency (C)} -- the replicated data is always consistent with each other;
        \item \textbf{Availability (A)} -- the data is highly available to the users;
        \item \textbf{Partition tolerance (P)} -- the system can continue providing services to its users even when the network partitions;
    \end{itemize}
\end{theorem}
%
\begin{note}
    The three properties are not exactly of the same sort, both technically and conceptually: consistency and availablity range over a spectrum of options, whereas partition tolerance can somehow be seen more as on/off feature.
All of them are \textit{desirable}, yet forfeiting partition tolerance is not really an option in real-world systems.
Particularly with pervasive systems in the IoT era, where \textit{instability}, of the network, rules.
\\
Long story short, CAP theorem most often forces us to \textit{choose between availability and consistency}
\end{note}
%
\begin{example}{Location-based games}{}
    \begin{itemize}
        \item \textbf{services} -- use information about the position of a user/device to provide information, entertainment, security;
        \item \textbf{games} -- use player's location to evolve and progress the gameplay;
    \end{itemize}
    Since the architecture is built around the notion off millions of players rooming physical space \textit{worldwide} with theri mobile devices, forfeiting \textbf{partition tolerance} is not an option:
    \begin{itemize}
        \item mobile devices are inherently \textit{unstable} in their network connectivity;
        \item players are inherently mobile, and so they can move in and out of network coverage;
        \item players tend to concentrate in some areas, e.g. during special events;
        \item scalability is a multi-level issue: not just the number of players overall, but also the number of places they can be at, and the number of players at each place;
    \end{itemize}
\textbf{Consistency} of the game data, like goals, situations, achievements, is essential to keep players going.
Location-based games are usually built around logically-centralized architecture; using spatial replication in order to reduce user-perceived latency and improve scalability.
When storing consistency is required, e.g. for in-game transactions, players might be forced to wait for the app servers to confirm the operation.
\end{example}

\subsection{CAP Proof}\label{sub:cap_proof} % (fold)
Brewer's first results required some \textit{proof}, possibly formal; for a proof, a more precise formulation of the conjecture is required.
\begin{cimage}
    \includegraphics[width=.5\textwidth]{assets/cap-proof.png}\label{fig:cap-proof}
    \caption{One way to see CAP}
\end{cimage}
\begin{definition}{Availability}{avlb}
    For a distributed system to be continously \textbf{available}, every request received by a non-failing node in the system must result in a response: if the system is available, we got \textit{responses}.
\end{definition}
\begin{definition}{Consistency}{conss}
    A \textbf{consistent} service is modelled as an \textit{atomic} data object, where:
    \begin{itemize}
        \item operations are totally ordered;
        \item each operation occurs in a single instant of time;
    \end{itemize}
    \begin{warning}
        Here, the meaning of \textit{consistent} is unlike \ac{acid} properties, since it encompasses both A and C there
    \end{warning}
    Among the many consequences, consistency implies that all \textit{read} operations over a distributed shared memory occurring after a \textit{write} operation completes must return the value of either this write operation or a later one. If the system is consistent, we got \textit{correct responses}
\end{definition}
\begin{definition}{Network partition}{net_part}
    When a network is partitioned, all messages sent from nodes in one component of the partition to nodes in another component are lost.
    And, any pattern of message loss can be medelled as a temporary partition separating the communicating nodes at the exact instant the message is lost.
\end{definition}
In short if node A send a message to node B, the network is \textit{partitioned} if the message does not take it to B.
\textit{Availability} is when B receives the message and responds.
\textit{Consistency} is when B response is correct.
\emptyln{}
The proof is given for three sorts of network:
\begin{itemize}
    \item synchronous network with message loss;
    \item synchronous network without message loss;
    \item partially synchronous network with local clocks;
\end{itemize}
For teaching purpose, we focus here on the \textit{asynchronous model}, where:
\begin{itemize}
    \item there is no single clock;
    \item nodes act based on local computation and message received;
\end{itemize}
\begin{theorem}{}{}
    It is impossible in the asynchronous network model to implement a read/write data object that guarantees the following properties:
    \begin{itemize}
        \item availability;
        \item atomic consistency;
    \end{itemize}
    in all fair executions (including those in which messages are lost)
    \begin{proof}
        \\
        \textbf{Assumptions}
        \begin{itemize}
            \item atomicity, availability and partition tolerance are all fulfilled;
            \item nodes in the network can be partitioned into two disjoint, non-empty sets $G_1$, $G_2$;
            \item atomic object $o$ has initial value $v_0$, $o$ is expected to be consistent through $G_1$, $G_2$;
            \item $\al_1$ executes a single write $v_1 \neq v_0$ of $o \in G_1$:
                \begin{itemize}
                    \item $\al_1$ is the only request in that time;
                    \item during $\al_1$ no message are received --- from $G_1$ to $G_2$, from $G_2$ to $G_1$;
                \end{itemize}
            \item no messages from $G_1$ are received in $G_2$;
            \item $\al_2$ executes a single read of $o \in G_2$: during $\al_2$ no message are received --- from $G_1$ to $G_2$, from $G_2$ to $G_1$;
        \end{itemize}
        Due to availability, $\al_1$ completes $\pt{v_0 \to v_1}$, and $\al_2$ does too.
        Executing $\al_1$ and $\al_2$, $G_2$ just sees $\al_2$, and must return $v_0$.
        \emptyln{}
        This obviously violates consistency as defined above, which is basically a sort of atomic consistency.
    \end{proof}
\end{theorem}
% subsection CAP Proof (end)

\subsection{Building on CAP}\label{sub:building_on_cap} % (fold)
Brewer himself later elaborated on his CAP theorem:
\begin{enumerate}
    \item when the network is partitioned, a distributed system should choose a tradeoff between consistency and availability;
    \item when there is no partition, a system can feature both consistency and availability
\end{enumerate}
\begin{note}
    \acf{acid} semantics might be too strong:
    \begin{itemize}
        \item The design space for network services can be partitioned according to the data semantics that each service demands.
        \item For other Internet services, however, the primary value to the user is not necessarily strong consistency or durability, but rather high availability of data;
    \end{itemize}
\end{note}
Stale data can be temporarily tolerated as long as all copies of data \textit{eventually} reach \textit{consistency} after a short time.
Soft state, which can be generated at the expense of additional computation or file I/O, is exploited to improve performance; data is not durable.
\textit{Approximate answers}, based on stale data or incomplete soft state, delivered quickly may be more valuable than exact answers.
%
\begin{definition}{BASE}{base}
    \begin{itemize}
        \item Basically Available;
        \item Soft state;
        \item Eventual consistency;
    \end{itemize}
\end{definition}
Most cloud services nowadays adopts BASE and try to mask inconsistencies from users; Steps ahead are leading to new models, based on, e.g. new data-consistency models.
\begin{note}[CAP Today]
    The modern CAP goal should be to maximise combinations of consistency ad availability that makes sense for the specific application.
    Such an approach incorporates plans for operation during a partition and for recovery afterward, thus helping designers think about CAP beyon its historycally perceived limitations.
\end{note}

\section{Dependability in Distributed Systems}\label{sec:dependability_in_distributed_systems} % (fold)
Distributed systems are playing an ever-increasingly important role in all aspects of our society, governaments, business, and individuals alike.
They are behind many services on whiche we depend on a daily basis, such as: financial, e-commerce, civil infrastructure, entertainment, personal data storage, etc.
Their \textit{dependability} matters to business as well as to every individual.
\\
The cost of system \textit{failure} is mostly and typically \textbf{enormous}; however also \textbf{dependability} is not cheap at all, which is possibly the reason why nearly 60\% of Fortune 500 companies suffer from more than one hour and a half of downtime per week.
\\
In order to reduce the cost of building and maintaining highly-dependable systems, we could try and train more experts that know how to design, implement and maintain dependable distributed systems.
%
\subsection{Dependability \& Faults}\label{sub:dependability_faults} % (fold)
\begin{quote}[Oxford Learner's Dictionaries]
    The quality of being able to be relied on to do what somebody wants or needs.
    \\
    \textbf{synonym} reliability
\end{quote}
The term ``dependable systems'', along with the related notions, belongs to the grneral scope of systems engineering, not just to computational systems --- let alone distributed ones.
In the context of \textit{distributed computing}, dependability refers to the ability of distributed system to provide correct services to its users despite the many different threats, like undetected software defects, hardware failures, malicious attacks\ldots
\\
In order to reason about dependability of distributed systems, we need a way to:
\begin{itemize}
    \item model a distributed system appropriately;
    \item model the \textbf{threats} to the system;
\end{itemize}
%

\subsection{System Models}\label{sub:system_models} % (fold)
System ModelsA distributed system is designed to provide a set of \textbf{services} to its users, often referred to as clients.
Each service has an \textbf{interface} that a client could use to request the service.
A \textit{functional specification} typically defines what a service should do, at each moment in time, a system is in a given \textbf{state}; what precisely is the state of a distributed system is a very complicated issue.
\begin{definition}{State}{}
    The \textbf{state} of a distributed system is determined \textit{collectively} by the state of the processes and threads in the system; it typically consists of the values of its registers, like stack, heap, file descriptors, kernel states etc.
Part of the state might become either implicitly or explicitly visible through user interaction, which we call \textbf{external state}, or observable state. Basically an abstract state as defined by the functional specification.
The remaining part of the state that is not visible to users, which we call \textbf{internal state}.
\end{definition}
Generally speaking, the state of a system at time $t$ is represented by the (minimum amount of) information that, along with the knowledge about the dynamics of a deterministic system, allows an observer to completely describe the future system behaviour --- from time $t$ on.
State can be used for \textbf{recovery} after a failure: a system can be recovered to the ``place'' where it was before a failure if its state was consistently captured and not lost due to the failure.
\begin{note}[How can we capture the state of a distributed system at time $t$?]
    First of all, we should define the \textbf{boundary} of a system that is, the physical/logical line that separates the system from its surrounding environment: inside the boundary the system's \textbf{components}, outside the boundary the system's \textbf{environment}.
    The term ``environment'' refers here to all other systems that the current system interact with --- in the very general sense that they affect the system in any possible way.
    Every definition of component vs. system is determined by the level of abstraction chosen, for example:
    \begin{itemize}
        \item at one give level of abstraction, we could observe a system, with its components, interacting with other systems;
        \item at another given level of abstraction, the same system could be instead described as a component of a larger system;
    \end{itemize}
\end{note}
However, this is not enough to assess that we know how to \textit{capture the state} of a distributed system. Mostly because we do not know (yet) what ``at time $t$'' would mean there: the complex issue of \textbf{time} in distributed systems.
%
% subsection System Models (end)
\subsection{Threat Models}\label{sub:threat_models} % (fold)
When a system is not compliant with its functional specification, we say that a \textbf{failure} has occurred.
The failure of a system is \textit{caused} by (part of) its state having wrong values --- i.e., \textbf{errors} in its state; the causal model assumes that errors are caused by some \textbf{faults}.
\begin{quote}[In other terms\ldots]
    A system is said to \textit{fail} (failure) when it does not behave as promised, an \textit{error} is a part of a system state that might have caused a failure and an error is a \textit{fault}.
\end{quote}
A fault might be \textbf{dormant}: it could not immediately show itself and cause error, like a bug in the software not causing problems until the corresponding code is executed, or a shared variable not lock protected in a multithreaded application until two or more threads try to update i concurrently.
\\
Given the typical recursive nature of system composition, the failure of one system may cause a fault in a larger system when the former constitutes a component of the latter.
Such a relationship between fault, error and failure is referred to as \textbf{\textit{chain of threats}}, which makes the terms ``fault'' and ``failure'' often used interchangeably in the literature --- somehow improperly.
\begin{cimage}[htp]
    \includegraphics[width=.8\textwidth]{assets/fault-and-failure.png}
    \caption{Relationship between faults and failures in larger systems}\label{fig:fault-and-failure}
\end{cimage}

\subsection{Faults}\label{sub:faults} % (fold)
Different sorts of faults require different treatment; faults can be classified according to different criteria:
\begin{itemize}
    \item the \textit{source} of the fault;
    \item the \textit{intent} of the fault;
    \item the \textit{duration} of the fault;
    \item the \textit{manifestation} of the fault;
    \item the \textit{reproducibility} of the fault;
    \item the \textit{relationship} of the fault with other faults;
\end{itemize}
\begin{definition}{Source of Faults}{}
    Based on their \textit{source}, faults can be classified as:
    \begin{itemize}
        \item \textbf{hardware faults}, if the faults are caused by the failure of hardware components such as power outages, hard drive failures, bad memory chips, etc.;
        \item \textbf{software faults}, if the faults are caused by software bugs such as reace conditions and no-boundary-checks for arrays;
        \item \textbf{operator faults}, if the faults are caused by the operator of the system, for example misconfiguration, wrong upgrade procedures, etc.;
\end{itemize}
\end{definition}
\begin{definition}{Intent of Faults}{}
    Based on their \textit{intent}, faults can be classified as:
    \begin{itemize}
        \item \textbf{non-malicious faults}, if the faults are not caused by a person with malicious intent, e.g. a naturally-occurred hardware fault, or some unintended software bugs; 
        \item \textbf{malicious faults}, if the faults are caused by a person with intent to harm the system, for example to deny services to legitimate clients or to compromise the integrity of the service;
\end{itemize}
\end{definition}
\begin{definition}{Duration of Faults}{}
    Based on their \textit{duration}, faults can be classified as:
    \begin{itemize}
        \item \textbf{transient faults}, if the fault is activated momentarily then goes dormant again, like when a power spike affects a hardware component then disappears;
        \item \textbf{intermittent faults}, if the fault occurs, vanishes of its own accord, then reappears, and so on, like race conditions occurring when and only when two threads access the same shared variable at the same time;
        \item \textbf{permanent faults}, if, once activated, the fault stays such unless the faulty component is repaired or the source of the fault is addressed: a power outage is considered a permanent fault because a computer system will remain powered off unless/until power is restored, or a crash fault;
    \end{itemize}
\end{definition}
\begin{definition}{Manifestation of Faults}{}
    Based on their \textit{manifestation}, faults can be classified as:
    \begin{itemize}
        \item \textbf{content faults}, if the faults cause the values passed on to other components to be wrong:
        \begin{itemize}
            \item a faulty component may always pass on the same wrong values to other components, or, it may return different values to different components that it interacts with;
            \item where the latter case is specifically modelled as \textit{Byzantine faults};
        \end{itemize}
        \item \textbf{timing faults}, if the faulty component either returns a reply too early, or too late after receiving a request from another component, an extreme case is when the faulty component stops responding at all (e.g., when the component crashes, or hangs due to an infinite loop or a deadlock);
\end{itemize}
\end{definition}
\begin{definition}{Reproducibility of Faults}{}
    Based on their \textit{reproducibility}, fault can be classified as:
    \begin{itemize}
    \item \textbf{reproducible faults} (\textit{deterministic} faults), if the fault happens deterministically and can be easily reproduced, like accessing a null pointer, these sorts of faults are typically easily identified and repaired;
    \item \textbf{nondeterministic faults}, if the faults appear to happen non-deterministically and is hard to reproduce, if a fault is caused by a specific interleaving of several threads when they access some shared variable, these sorts of faults are also referred to as \textit{Heisenbugs} to highlight their uncertainty;
\end{itemize}
\end{definition}
\begin{definition}{Relationship of Faults}{}
    Based on their \textit{relationship} with other faults, faults can be classified as:
    \begin{itemize}
        \item \textbf{independent faults}, if there is no causal relationship between the faults: given fault $A$ and fault $B$, $B$ is not caused by $A$, and $A$ is not caused by $B$;
        \item \textbf{correlated faults}, if the faults are causally related: given fault $A$ and fault $B$, either $B$ is caused by $A$, or $A$ is caused by $B$; if multiple components fail due to a common reason, the failures are referred to as \textit{common mode} failures;
    \end{itemize}
\end{definition}
\begin{cimage}
    \includegraphics[width=.8\textwidth]{assets/failure-models-classification.png}
    \caption{Another classification of Failure Models}\label{fig:failure-models-classification}
\end{cimage}
%
\newpage
\begin{definition}{Fail-Stop System}{}
    When the system fails, it is desirable to avoid \textit{catastrophic consequences}, the consequences of the failure of a system can be alleviated by enhancing the system with dependability mechanisms such that when it fails, it stops responding to requests.
    Such systems are referred to as \textbf{fail-stop systems}.
\end{definition}
\begin{definition}{Fail-Safe System}{}
    Sometimes, a system is such that its failure does not cause great harm either to human life or the environment, such systems are referred to as \textbf{fail-safe systems}.
    Usually a fail-safe system defines a set of \textit{safe states}, when a fail-safe system can no longer operate according to its specification, it can transit to one of the predefined safe states, e.g., the computer system used to control a nuclear power plant must be a fail-safe system.
\end{definition}
\begin{definition}{Fail Fast}{}
    It is often desirable for a system to halt its operation immediately when it is an error state or encounters an unexpected condition. The software engineering practice to ensure such a behaviour is called \textbf{fail fast}.
    The fail-fast practice enables \textit{early detection} of software faults and the diagnosis of faults, given that when a fault has been propagated to many other components, it way harder to pinpoint the source of the problem.
\end{definition}

\subsection{Dependability Attributes and Evaluation Metrics}\label{sub:dependability_attributes_and_evaluation_metrics} % (fold)
The main features of dependable systems are:
\begin{itemize}
    \item availability;
    \item reliability;
    \item safety;
    \item maintainability;
\end{itemize}
Like in \Cref{def:avlb}, availability refers to the property that a system is ready for immediate use.
This means that availability refers to the \textit{probability} that a system is operating correctly at any given moment, ready to provide users with its functions.
So, a \textit{highly-available system} is a system that is most likely to be ready and working at any given instant of time.
\begin{definition}{Reliability}{}
    \textbf{Reliability} refers to the property that a system can run continuously without a failure.
    This means that reliability is defined in terms of a \textit{time interval}, rather than of an instant, so a \textit{highly-reliable} system is a system that is most likely to keep on running for a long period of time.
\end{definition}
\begin{definition}{Safety}{}
    \textbf{Safety} refers to the situation that when a system temporarily fails to operate correctly, nothing \textit{catastrophic} happens.
    This is a very difficult property to be defined, and to be ensured as well; when \textit{catastropich} basically refers to those states of affairs when there is no established path from fault back to normal operation.
\end{definition}
\begin{definition}{Maintainability}{}
    \textbf{Maintainability} refers to how easily a failed system can be repaired.
    This means that maintainability is closely related to availability, so a \textit{highly-maintainable} system may also show a high degree of availability.
\end{definition}
%
\subsubsection{Attributes of a Dependable System}\label{sec:attributes_of_a_dependable_system} % (fold)
A dependable system has a number of \textit{desirable \textbf{attributes}}:
\begin{itemize}
    \item availability;
    \item reliability;
    \item integrity;
    \item maintainability;
    \item safety;
\end{itemize}
\begin{quote}
    Some attributes can be used as \textit{evaluation metrics} for the system, like availability and reliability; while the others are difficult to quantify.
    \\
    Some attributes are \textit{fundamental} to all distributed systems, like availability, reliability and integrity; while the others are secondary or not applicable to all systems.
\end{quote}
% subsubsection Attributes of a Dependable System (end)
% subsection Dependability Attributes and Evaluation Metrics (end)
\subsection{Availability}\label{sub:availability} % (fold)
Availability is a measure of the readiness of a dependable system at a(ny) point in time, i.e., when a client needs to use a service provided by the system, the probability that the system is there to provide the service to the client.
\begin{note}[Availability Factors]
    \begin{itemize}
        \item \ac{mttf}, the average time until a system fails;
        \item \ac{mttr}, the average time needed to repair a system;
        \item \ac{mtbf}, just $MTTF + MTTR$;
    \end{itemize}
\end{note}
\begin{equation*}
    \text{Availability} = MTTF / \pt{MTTF + MTTR} = MTTF / MTBF
\end{equation*}
The availability of a system is typically represented in terms of \textit{how many 9s.}
For example, if a system is claimed to offer \textit{five 9s availability}, it means that the system will be available with a probability of $99,999\%$.
If the system has $\power{10}{-5}$ probability to be unavailable when a client tries to access the service at any time --- which means that the system may have at most $5,256$ minutes of downtime a year.
% subsection Availability (end)
\subsection{Reliability}\label{sub:reliability} % (fold)
Reliability is a measure of the system's capability of providing correct services \textit{continuously} for a period of time.
It is proportional to \ac{mttf}, and it's often represented as the probability for the system to do so for a given period of time $\Dt t$.
\begin{equation*}
    R\pt{\Dt t} = \power{e}{-\lb \Dt t} \quad \lb \in \itro{0, \ift}
\end{equation*}
\begin{warning}[availability $\ne$ reliability]
    A system failing frequently but recovering very quickly may have high availability, and at the same time very low reliability.
\end{warning}
% subsection Realiability (end)
\subsection{Integrity}\label{sub:integrity} % (fold)
\begin{definition}{Integrity}{}
    Refers to the capability of a system to protect its state from being compromised under various threats.
\end{definition}
In dependable computing research, integrity typically translates into the consistency of server replicas, if redundancy is exploited.
As long as the number of faulty replicas does not exceed a pre-defined threshold, the consistency of the correct replicas naturally implies system integrity.
% subsection Integrity (end)
\subsection{Maintainability}\label{sub:maintainability} % (fold)
Maintainability refers to the capability of a system to evolve after it is deployed.
Once a software fault is detected, applying a repairing patch should not involve uninstalling the existing software and reinstalling an updated one. The same patching/software update mechanism may be used to add new features or improve the performance of the existing system.
\begin{quote}
    Ideally a highly-maintainable system is easy to repair/improve (live upgrade)
\end{quote}
% subsection Maintainability (end)
\subsection{Safety}\label{sub:safety} % (fold)
Safety means that when a system fails, it does not cause \textit{catastrophic consequences.}
The system must be fail-safe: system that are used to control operations that may cause catastrophic consequences, such as nuclear power plants, or endanger human lives, such as hospital operation rooms, must bear the safety attribute.
Safety attribute is not important for systems not operating in such environments, e.g., e-commerce.
% subsection Safety (end)

\section{Means to Achieve Dependability}\label{sec:means_to_achieve_dependability} % (fold)
\subsection{Fault Avoidance}\label{sub:fault_avoidance} % (fold)
Achieved by building and using quality software components and hardware that are less prone to failures.
As far as software is concerned, fault avoidance aims at ensuring correct design specification and correct implementation before a distributed system is deployed:
\begin{itemize}
    \item \textit{rigorous software design}, possibly with the use of formal method to assess and verify the system properties;
    \item \textit{rigorous software testing}, to identify and remove software bugs due to design deficiency or introduced during implementation;
\end{itemize}
% subsection Fault Avoidance (end)
\subsection{Fault Detection \& Diagnosis}\label{sub:fault_detection_diagnosis} % (fold)
The \textit{detection} can be tricky, crash faults are trivial to detect: we can periodically probe each component to check on its health, however components in a practical system might fail in various ways other than crash implementation, so probing does not always help.
\begin{note}
    If a fault is not detected, the integrity of the system cannot be guaranteed.
\end{note}
Once the fault is detected, \textit{diagnosis} is required to determine that a fault indeed has occurred, and to localize the source of the fault.
\begin{example}{Exception Handling}{}
    An example of fault detection and handling in represented by \textit{exception handling} in modern programming languages.
\end{example}
% subsection fault Detection \& Diagnosis (end)
\subsection{Fault Removal}\label{sub:fault_removal} % (fold)
Once a fault is detected and localized, it should be isolated and removed from the system.
Then the faulty component is either repaired or replaced, and reintroduced in the system, which typically requires reconfiguration.
In a distributed system, this typically requires a notion of \textit{membership}:
\begin{itemize}
    \item the faulty component is excluded from the system;
    \item the repaired component becomes part of the membership again;
\end{itemize}
\begin{quote}
    Software updates are special cases
\end{quote}
% subsection Fault Removal (end)
\subsection{Fault Tolerance}\label{sub:fault_tolerance} % (fold)
Because of hardware failures, robust software itself is not enough to delivery high dependability, unless a system is fully stateless: simply restarting after a failure would not automatically restore its state before the failure.
Hence, fault tolerance techniques are essential to improve the dependability of distributed systems to the next level.
\begin{note}[Fault tolerance techniques vs.\ dependability requirements]{}
    Different fault tolerance techniques can be used to target different levels of dependability requirements.
    For applications that need high availability, but not necessarily high reliability, \textbf{logging} and \textbf{checkpointing}, could be enough; more demanding applications could adopt \textbf{recovery oriented} computing techniques.
    Both classes of techniques rely on \textbf{rollback recovery} --- that is, back to the most recent recorded correct state
\end{note}
\begin{example}{Masking Failure by Redundancy}{}
    \textbf{Idea:} hiding failures from other processes, the key technique for masking faults is \textbf{redundancy.}
    Type of redundancy:
    \begin{itemize}
        \item \textbf{information}, extra bits;
        \item \textbf{time}, redos after transaction aborts;
        \item \textbf{physical}, typical in biological systems;
    \end{itemize}
\end{example}

\subsection{Basic Techniques for Dependability}\label{sub:basic_techniques_for_dependability} % (fold)
\textbf{Checkpointing} and \textbf{logging} are the most fundamental techniques to achieve dependability in distributed systems, by providing a path towards \textit{distributed system recovery} after failure.
By themselves, they provide a form of fault tolerance that is relatively easy to implement and incurs low runtime overhead.
They have limitations, mostly when they are used by themselves, e.g, some information could be lost, if only checkpointing is used, when a fault occurs, also the recovery time after a fault is typically larger than that of more sophisticated fault tolerance approaches.
However, they are both used in \textit{all levels} of dependability mechanisms.
\begin{definition}{Checkpointing}{}
    A \textbf{checkpoint} of a distributed system refers to a \textbf{copy of the system state}, if the checkpoint is available after the system fails, it can be used to \textit{recover the system} to the state when the checkpoint was taken.
    Technically, checkpointing refers to the action of, periodically, taking a copy of the system state, and \textit{saving} the checkpoint to a stable storage that can survive the faults tolerated.
\end{definition}
To recover the system to the point right before it fails, other recovery information must be logged \textit{in addition} to periodical checkpointing, to this end all incoming messages to the system are typically logged.
Also other non-deterministic events may have to be logged as well to ensure proper recovery.
Checkpointing and logging provide a form of \textit{rollback recovery}, since they can recover the system to a state prior to the failure.
% subsection Basic Techniques for Dependability (end)

\subsubsection{Fault Model}\label{sec:fault_model} % (fold)
\textbf{Assumptions}:
\begin{itemize}
    \item failure occurs in a process;
    \item when a process fails, it simply execution and loses all its volatile state --- fail-stop;
    \item communication is reliable --- e.g, \ac{tcp} --- and \ac{fifo};
    \begin{itemize}
        \item message ordering maintained;
        \item no network partition;
    \end{itemize}
\end{itemize}

\subsubsection{Process State}\label{sec:process_state} % (fold)
\textbf{Assumptions}:
\begin{itemize}
    \item the state of each individual process is defined by its entire address space in the operating system;
    \begin{itemize}
        \item a generic checkpointing library just saves the entire address space as a checkpoint of the process;
    \end{itemize}
    \item the specific nature of the process and the application semantics can be exploited to define a smaller and more specific notion of process state;
\end{itemize}
% subsubsection Process State (end)

\subsubsection{Global State}\label{sec:global_state} % (fold)
The state of a distributed system is usually referred to as the \textbf{global state} of the system: which is meant to include the state of every process in the system.
Aggregation of process state is not enough: the states of different processes in a distributed system are \textit{related}:
\begin{itemize}
    \item because message exchanges, moving information between processes;
    \item information exchange causing change of state;
    \item processes causally depend on other processes;
    \item process states are \textit{interdependent};
\end{itemize}
\begin{critical}
    Dependencies cannot be lost in the global state
\end{critical}
\begin{cimage}
    \includegraphics[width=.8\textwidth]{assets/global-states-type.png}\label{fig:global-states-type}
\end{cimage}
\begin{example}{Inconsistent Global State}{}
    Checkpoints taken by different processes are \textit{incompatible}, they cannot be used to recover the system upon failure.
    In fact: $C_1$ reflects reception of $m_0$, $C_0$ does not.
    For example, $P_0$ and $P_1$ represent two bank accounts $A$ and $B$, respectively, $m_0$ represents a $100\$$ deposit from $A$ to $B$; if $P_0$ crashes after $m_0$, and $P_1$ after checkpoint $C_1$, recovery from $C_0$ and $C_1$ would make $100\$$ appear from nowhere.
    Global state from the wrong set of checkpoints is not \textit{admissible}, i.e., it's not reachable from the initial state of the system.
    This is what is called an \textbf{inconsistent global state}.
\end{example}
\begin{example}{Consistent Global State}{}
    Checkpoints taken by different processes are \textit{compatible}, they could be used to recover the system upon failure.
    In fact: $C_1$ reflects reception of $m_0$, so does $C_0$, the same holds for $C_1$, $C_2$, and $m_1$; e.g., $P_0$ and $P_1$ represent two bank accounts $A$ and $B$, respectively, $m_0$ represents a $100\$$ deposit from $A$ to $B$, if $P_0$ crashes after $C_0$, and $P_1$ after $C_1$, recovery from $C_0$ and $C_1$ would make $100\$$ correctly move from $A$ to $B$.
\end{example}
\begin{example}{Consistent yet Unrecoverable Global State}{cons_unr_state}
    Checkpoints taken by different processes are \textit{compatible}, yet they could not be used to recover the system upon failure.
    In fact: checkpoints are consistent, since they represent a reachable state for the system, yet \textit{dependencies} on $m_0$ and $m_1$ would be lost on recovery; e.g., in the example, $100\$$ would possibly disappear.
    The problem is the loss of the messages in transit, in order to accommodate this scenario, a further sort of state is required: \textbf{channel state}.
\end{example}
% subsubsection Global State (end)
\subsubsection{System Model Refined}\label{sec:system_model_refined} % (fold)
A set of $N$ processes:
\begin{itemize}
    \item a process consists of a set of states and a set of events;
    \item one state is the initial state;
    \item events trigger the change of state of a process;
\end{itemize}
A set of channels:
\begin{itemize}
    \item a channel is a uni-directional reliable communication channel between two processes;
    \item the state of a channel is the set of messages that are in transit along the channel --- that is, not yet received by the target process;
    \item a \ac{tcp} connection is two channels;
\end{itemize}
In scenario \cref{ex:cons_unr_state}, if $m_0$ is saved in $C_0$ as channel state, and $m_1$ in $C_1$ the same way, recovery from $C_0$ and $C_1$ is possible.
% subsubsection System Model Refined (end)
\begin{definition}{Piecewise Deterministic Assumption}{}
    Checkpoint-based protocols only ensure to recover the system up to the most recent consistent global state that has been recovered, and all executions happened afterward are lost.
    Instead, logging (log-based protocols) can be used to recover the system to the state right before the failure, provided that all events are logged, and the log is available upon recovery.
    They work based on a model assumption, referred to as the \textbf{piecewise deterministic assumption}, that is:
    \begin{itemize}
        \item each state evolves deterministically until a nondeterministic even happens, such as the reception of a message;
        \item all nondeterministic events can be identified;
        \item enough information is logged for each event;
    \end{itemize}
\end{definition}
\subsubsection{Output Commit}\label{sec:output_commit} % (fold)
A distributed system interacts with the outside world, such as the clients of the services provided by the distributed system.
Once output is emitted, a portion of the state of the distributed system becomes \textit{observable} by the outside world --- as a sort of commitment, or achievement.
Recovery should also account for a \textit{observably-consistent view} of the system from the outside.
Even though, if a failure occurs, the outside world cannot be relied upon recovery: this is the \textbf{output commit problem}.
\begin{warning}
   Observable consistency requires that enough recovery information is logged before the system commits to an output message. 
\end{warning}
% subsubsection Output Commit (end)
\subsubsection{Stable Storage}\label{sec:stable_storage} % (fold)
An essential requirement for logging and checkpointing protocols is the availability of \textbf{stable storage}, stable storage can survive process failures in that upon recovery.
The information stored in the stable storage is readily available to the recovering process, as such, all checkpoints and messages logged must be stored in stable storage.
There are various forms of stable storage:
\begin{itemize}
    \item to tolerate only process failures, local disks are enough;
    \item to tolerate disk failures, redundant disks --- e.g., RAID-1 or RAID-5 --- could be used as stable storage;
    \item replicated file systems, or cloud-based file systems can be used as more robust stable storage;
\end{itemize}

\subsection{Checkpoint-Based Protocols}\label{sub:checkpoint_based_protocols} % (fold)
Focussing on a state, rather than on events, checkpoint-based protocols do not rely on the piecewise deterministic assumption: they are simpler to implement and less restrictive --- since there is no need to identify all forms of nondeterministic events and log them properly.
However, if we cannot take a checkpoint before every event, loss of execution has to be tolerated, and taking a checkpoint before every event is typically not realistic.
%
\subsubsection{Uncoordinated Checkpointing}\label{sec:uncoordinated_checkpointing} % (fold)
In \textit{uncoordinated checkpointing}, each process in the distributed system is autonomously decided when to checkpoints.
The intuition is that, when a failure occurs, the recovery processes goes backwards to select the most recent set of consistent checkpoints for the recovery of the global state.
The main problem is that the checkpoints taken by the processes might not allow reconstruction of a consistent global state.
\\
To enable the selection of a set of consistent checkpoints during recovery, the \textit{dependency} between the checkpoints has to be determined and recorded together with each checkpoint.
\begin{warning}{}{}
    This incurs additional overhead and increases the complexity of the implementation
\end{warning}

\subsubsection{Tamir and Sequin Checkpointing Protocol}\label{sec:tamir_and_sequin_checkpointing_protocol} % (fold)
The Tamir and Sequin global checkpointing protocol is a \textit{coordinated checkpointing protocol}, where:
\begin{itemize}
    \item one of the processes is designated as the \textbf{coordinator}, and knows all the other processes in the system;
    \item the remaining processes are \textbf{participants};
    \item the coordinator uses a \textbf{two-phase commit protocol} to ensure that:
    \begin{itemize}
        \item the checkpoints taken at individual processes are consistent with each other;
        \item the global checkpointing operation is either carried out as an atomic transaction or aborted --- that is, either all processes successfully create a new set of checkpoints or they abandon the current round and revert to their previous set of checkpoints;
    \end{itemize}
    \item first phase aims at creating a \textit{quiescent} point of the distributed system state;
    \item second phase aims at atomically moving from the old checkpoint to the new one;
    \item checkpointing round is aborted when a participant fails to respond;
\end{itemize}
\begin{note}[Control Messages]{}
    \begin{itemize}
        \item \textbf{CHECKPOINT} message --- used to initiate a global checkpoint, also used to establish a quiescent point of the distributed system where all processes have stopped normal execution
        \item \textbf{SAVED} message --- used to initiate a global checkpoint, also used to establish a quiescent point of the distributed system where all processes have stopped normal execution
        \item \textbf{FAULT} message --- used for participant to inform the coordinator that it has done a local checkpoint;
        \item \textbf{RESUME} message --- used by the coordinator to inform the participants that they now can resume normal execution;
    \end{itemize}
\end{note}
\begin{cimage}
    \includegraphics[width=.6\textwidth]{assets/finite-state-machine-coordinator.png}
    \caption{Finite State Machine for Coordinator}\label{fig:finite-state-machine-coordinator}
\end{cimage}
%
\begin{note}[Coordinator Behaviour]{}
    \begin{itemize}
        \item starts first phase by stopping normal execution and sending CHECKPOINT message to every process;
        \item waits for CHECKPOINT incoming messages from all processes --- aborts when some are missing;
        \item checkpoints its state;
        \item waits for SAVED messages from every process --- aborts when some are missing;
        \item sending FAULT messages, switches to the new checkpoint, and sends RESUME message to every process;
        \item resumes normal execution; 
    \end{itemize}
\end{note}
\begin{cimage}
    \includegraphics[width=.8\textwidth]{assets/finite-state-machine-participant.png}
    \caption{Finite State Machine for Participant}\label{fig:finite-state-machine-participant}
\end{cimage}
\begin{note}[Participant Behaviour]{}
    \begin{itemize}
        \item stops normal execution upon CHECKPOINT message reception;
        \item sends CHECKPOINT messages through all outgoing channels and waits for CHECKPOINT from all incoming channels --- aborts when some are missing;
        \item checkpoints its state;
        \item passes SAVED message from one of its downstream neighbour processes up to its upstream neighbour;
        \item propagates RESUME message through every outgoing channels --- except the one that just sent RESUME;
        \item resumes normal execution;
    \end{itemize}
\end{note}

\subsubsection{Chandy and Lamport Distributed Snapshot Protocol}\label{sec:chandy_and_lamport_distributed_snapshot_protocol} % (fold)
The Tamir and Sequin global checkpointing protocol is a \textit{blocking protocol}: normal execution is suspended during each round of global checkpointing.
The Chandy and Lamport \textbf{distributed snapshot protocol} is a non-blocking protocol: normal execution is not interrupted by the global checkpointing.
However, unlike the Tamir and Sequin protocol, the Chandy an Lamport protocol is only concerned with how to produce a consistent global checkpoint.
It prescribes no mechanism on how to determine the end of the checkpointing round, and how to atomically switch over to the new global checkpoint.
\begin{cimage}
    \includegraphics[width=.6\textwidth]{assets/chandy-lamport-snapshot-protocol.png}
    \caption{Chandy and Lamport Distributed Snapshot Protocol}\label{fig:chandy-lamport-snapshot-protocol}
\end{cimage}
\begin{note}[Protocol Description]{}
    Processes can be either \textit{Normal} or \textit{Checkpointing} state.
    Any process can initiate global checkpointing, taking a local checkpoint and sending a Marker message to all outgoing channels --- state to Checkpointing.
    Upon Marker message reception, a process moves from Normal to Checkpointing state, does the same as above plus records the Marker Certificate --- which needs to be completed, when the Marker Certificate is completed, a process goes back into Normal state.
\end{note}
\begin{cimage}
    \includegraphics[width=.8\textwidth]{assets/chandy-lamport-system-topology.png}
    \caption{Chandy and Lamport System Topology}\label{fig:chandy-lamport-system-topology}
\end{cimage}

\begin{quote}[Analogies]{}
    Both protocols rely on basically the same system model, and use a special control message to propagate and coordinate the global checkpointing.
    They both recognize the need to capture the channel state to ensure the recoverability of the system, the mechanism to capture the channel state is virtually the same for both protocols.
    The communication overhead of the two protocols is identical.
\end{quote}
\begin{quote}[Differences]{}
    \begin{itemize}
        \item the blocking protocol is more conservative;
        \item the blocking protocol is more complete and robust;
        \item the non-blocking protocol provides no mechanism for the atomicity of the global checkpointing round;
        \item the blocking protocol leverages on a coordinated checkpointing protocol, whereas the non-blocking one promotes autonomy of the protocol initiation --- which may fit differently diverse sorts of distributed systems;
    \end{itemize}
\end{quote}

\subsection{Log-Based Protocols}\label{sub:log_based_protocols} % (fold)
Checkpoint-based protocols leverage on saving state and recovery from there, at the possible expense of some loss of execution.
Logging leverage on saving events, which can be used to recover the system to the state right before the failure, provided that the \textit{piecewise state intervals} --- intervals between (non-deterministic) events.
Each state interval is initiated by a non-deterministic event --- such as the receiving of a message --- or the initialization of the process, and followed by a sequence of deterministic state changes.
As long as the non-deterministic event is logged, the entire state interval can be replayed.
Sorts of logging protocols:
\begin{itemize}
    \item \textbf{pessimistic} logging --- message received is synchronously logged prior to its execution;
    \item \textbf{optimistic} logging --- to reduce the latency overhead, the non-deterministic events are first stored in volatile memory, then logged asynchronously to stable storage: the failure of a process might result in permanent loss of some messages, which would force a rollback to a state earlier than the state when the process fails;
    \item \textbf{causal} logging --- the non-deterministic events not yet logged to stable storage are piggybacked with each message sent. With the piggybacked information, a process can have access to all the non-deterministic events that may have causal effects on its state, thereby enabling a consistent recovery of the system upon a failure;
\end{itemize}
In both optimistic and casual logging protocols, the dependency of the processes has to be tracked, and sufficient dependency information has to be piggybacked with each message sent.
This not only increases the complexity of the logging mechanisms, but most importantly, makes the failure recovery more sophisticated and expensive because the recovering process has to find a way to examine its logs and determines if it is missing any messages and often causes cascading recovery operations at other processes.
\\
Pessimistic logging protocols are much simpler in their design and implementation and failure recovery can be made much faster.
\emptyln{}
For all practical purposes, logging is always used in conjunction with checkpointing, so that states are saved as checkpoints, and events are logged from one checkpoint to the next one.
In this way there are two benefits:
\begin{itemize}
    \item limiting the recovery time because to recover from a failure in the process can be restarted from its last checkpoint (instead from its initial state), and its state can be recovered prior to the failure by replaying the logged non-deterministic events;
    \item limiting the size of the log: by taking a checkpoint periodically, the logged events prior to the checkpoint can be garbage collected;
\end{itemize}

\section{Replication \& Consistency in Distributed Systems}\label{sec:replication_consistency_in_distributed_systems} % (fold)

\subsection{Why Replication?}\label{sub:why_replication_} % (fold)
The replication of data:
\begin{itemize}
    \item increases the reliability of systems;
    \item improves preformance;
    \item allows scaling in numbers and in geographical area;
\end{itemize}
\begin{quote}[Replication for Reliability]{}
    If a file system has been replicated, system operation may continue after one replica crashes by simply switching to one of the other replicas.
    Maintaining multiple copies allows for better protection against corrupted data: if we have three copies of a file, and perform every read and write operation on each copy, we could hide a single failing write operation by considering as correct the value returned by at least two copies.
\end{quote}
\\
\begin{quote}[Replication for Performance]{}
    Replication for performance is important when a distributed system needs to scale in terms of size or the geographical area covered:
    \begin{itemize}
        \item scaling regarding size occurs when an increasing number of processes needs to access data managed by a single server. Performance can be improved by size replicating the server, and geographical area subsequently dividing workload among the processes accessing the data;
        \item when scaling over a geographical area, time to access data decreases by placing a copy of data near to the process using them: performance as perceived by that process increases, but keeping all replicas up to date may consume more network bandwidth;
    \end{itemize}
\end{quote}
